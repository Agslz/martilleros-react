================================================================================
BACKEND – QUÉ AGREGAR PARA SUBASTAS E IMÁGENES DEL MATRICULADO
================================================================================

El frontend ya usa estos dos flujos:

  1) Crear subasta (un solo request)
     POST /api/private/subastas  [Bearer]
     Body JSON: SubastaRequest (titulo, descripcion, precioInicial,
       martilleroACargo, nombreMartillero, cuitMartillero, domicilio,
       fechaInicio, fechaFin).
     Response: ApiResponse<SubastaResponse> (con id de la subasta creada).

  2) Subir imágenes (un request por imagen, después de crear la subasta)
     POST /api/private/subastas/{id}/imagenes  [Bearer]
     Body: multipart/form-data
       - file (obligatorio): archivo imagen (ej. image/jpeg, image/png, image/webp).
       - orden (opcional): número para orden (1, 2, 3...).
     Response: ApiResponse<FileUploadResponse> con { fileName, fileUrl }.

================================================================================
PASOS EN EL BACKEND (Spring Boot)
================================================================================

1) POST /api/private/subastas [Bearer]

   - Seguridad: la ruta debe estar bajo /api/private/** (ya tiene hasAnyRole(ADMIN, MATRICULADO)).
   - Controller: por ejemplo SubastaController o un nuevo método en el controller de subastas.
   - Lógica:
     - Obtener el matriculado del JWT (matrícula del subject o del SecurityContext).
     - Buscar el Matriculado en BD por matrícula.
     - Crear la entidad Subasta con los datos del body y asignar el martillero
       (martilleroACargo = matrícula del usuario, nombreMartillero y cuitMartillero
       del usuario o del body).
     - Guardar y devolver SubastaResponse (con id).
   - Validar que solo un matriculado pueda crear para sí mismo (o que el backend
     ignore martillero del body y siempre use el del JWT).

   Ejemplo de ruta en SecurityConfig (si no está ya cubierta):
   - /api/private/** ya está permitido para ADMIN y MATRICULADO; solo hace falta
     que el controller exponga POST /api/private/subastas y que el servicio
     cree la subasta asociada al usuario del token.

2) POST /api/private/subastas/{id}/imagenes [Bearer]

   - Seguridad: misma regla /api/private/**.
   - Controller: método que reciba @PathVariable Long id, @RequestParam("file") MultipartFile file,
     @RequestParam(value = "orden", required = false) Integer orden.
   - Lógica:
     - Cargar la Subasta por id.
     - Verificar que la subasta pertenezca al matriculado logueado (por ejemplo
       que subasta.getMartilleroACargo() coincida con la matrícula del JWT, o
       que tengas una relación Subasta -> Matriculado y comparar con el usuario actual).
       Si no es suya, devolver 403.
     - Guardar el archivo (StorageService), crear ImagenSubasta asociada a la subasta
       con fileName, fileUrl, orden.
     - Devolver FileUploadResponse (fileName, fileUrl).

   Importante: solo permitir subir imágenes a subastas que pertenezcan al usuario
   logueado (matriculado), para no permitir que un usuario suba fotos a la subasta de otro.

================================================================================
RESUMEN DE ENDPOINTS A TENER EN EL BACKEND
================================================================================

  POST /api/private/subastas
    - Body: SubastaRequest (JSON).
    - Auth: Bearer (matriculado o admin).
    - Crea la subasta y asigna martillero desde el usuario logueado (o desde el body si es admin).
    - Response: SubastaResponse con id.

  POST /api/private/subastas/{id}/imagenes
    - Body: multipart (file, orden opcional).
    - Auth: Bearer.
    - Comprobar que la subasta {id} sea del usuario (matrícula) antes de subir.
    - Response: FileUploadResponse.

Si el backend ya tiene POST /api/admin/subastas y POST /api/admin/subastas/{id}/imagenes,
podés reutilizar la misma lógica de negocio (crear subasta, guardar imagen) y en el
controller exponer además las rutas bajo /api/private/ que deleguen al mismo servicio,
validando en el servicio que el usuario sea el dueño de la subasta cuando se use la ruta
privada.
